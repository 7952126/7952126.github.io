Collection 体系

Collection 是什么

  1. Collection 可以看作一个接口，类似多态，是不能被实例化的（new Collection）。

  2. 可以将它具体化的理解为：一篮子鸡蛋，一个队列的人等。它只包含一种元素。

  3. 作用：批量操作成千上万的数据。

  4. Collection 体系有两个很重要的接口：Set，List。

List/Set 约定

  1. List
  • List 是个有序的集合，是一个个有顺序的元素，可以拓展 Collection 体系。
  • List 有一些其他方法，比如 get，可获取指定位置的元素。
  • List 中有 n 个元素的时候，会将第一个元素约定为第 0 个，最后一个元素就是第 n-1 个。
  • 在 list 上加个泛型 list<>，它就会变成一个有类型的 list。

ArrayList 基于数组的 list。

  • ArrayList 是需要给定初始容量的。
  • 动态扩容：每当发现 list 放不下的时候，就通过算法来扩容
  1. Set
  • set 是个无序且不允许出现重复的集合(判断方法：equals)。

linkedHashSet 和 HashSet 和 TreeSet

  • HashSet 的顺序是完全随机的

  • linkedHashSet 保证输出的顺序和插入的顺序一样

  • TreeSet 它是有序的，使用 Comparable 约定。它就是红黑树（二叉树的一种）

Java 世界第二重要的约定:HashCode

  1. 对任意对象映射成 int，直接调用 HashCode 方法，可以有效降低查找的次数，提高查找的效率。

  2. 是一个单向的映射，不能返回来。

  3. 同⼀个对象必须始终返回相同的 hashCode • 两个对象的 equals 返回 true，必须返回相同的 hashCode：两个对象相等的话，他们的 HashCode 一定是相等的。 • 两个对象不等，也可能返回相同的 hashCode

  4. HashSet 的缺点：不保证顺序；如果想实现排序，使用 LinkedHashSet。

  5. HashCode 和 HashMap 这两者本质上是一种东西：HashSet 的 key 的 Set，就是一个 HashMap。

Map

  1. 把键映射到值的一种方式，一个键对应一个值，但反过来不一定能够成立。

  2. 一个 key 首先映射到一个 value，再生成 HashCode，然后再被丢到对应的 Hash 桶中。这个桶里面就有了许许多多一模一样 HashCode 的元素。

Collections

  • 用于存放 Collection 的各种工具方法

  • 例如：unmodifiableSet，将当前的 Set 变为不可修改的。

线程不安全性

HashMap 的实现不是线程安全的。

  • HashMap 在扩容的时候，可能会变成死循环。

  • 如果要在多线程环境下使用 HashMap，应该用 Concurrent HashMap（并发的 HashMap）

链表，红黑树

  • 这个 Java7 到 Java8 的变化。

  • 假如说传入的所有数据都正好拥有同一个 HashCode，这时候 HashMap 就退化成了一个链表，这时候，这一个桶的性能就会急剧恶化，会发现 HashMap 的性能下降了。因为 Hash 桶存在的意义是希望这些元素平均分配在不同的桶中，但没能实现。

  • 因此，在 JDK7 之后，当发生这种情况后，链表会变成红黑树（较复杂）。

  • 红黑树的出现是因为 ArrayList 的查找效率太低。
